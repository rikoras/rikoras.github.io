Empowering 1000 tokens/second on-device LLM prelling with mllm-NPU

关键词：LLM，端侧推理，NPU，软硬件协同，针对NPU的量化

Introduction

先提及了在某些任务中，prefill阶段占整个任务的绝大多数时间（dominates the end-to-end inference latency）（例如，提取屏幕中的信息作为上下文进行回复，prompt会非常长，600-800个token）。此处@powerinfer，动态激活稀疏性只能对decoding阶段的gemv操作有加速，无法处理这种场景。投机解码的方法也仅对decoding阶段有加速。

prefilling是计算密集型操作，对手机的cpu和gpu不友好（缺乏并行性能）。

NPU的优势在于强大的并行性能和优秀的能耗。

接着，列出三点挑战，指出直接用NPU存在的问题：（文中的说法是NPU和LLM存在semantic gap）

  1. NPU主要用于处理静态shape。但是prompt的长度是动态的。重新构建计算图开销太大（数秒甚至更多）。使用padding又造成算力浪费。

  2. 目前绝大多数NPU不支持分组量化的matmul。也即，模型权重分为若干个组，不同组使用不同的量化scale，可以避免将离群值截断造成大的性能损失。如果分成多个matmul，则NPU需要多做很多次reduce的浮点加法，造成10倍以上的性能损失。

  3. 接上一条，NPU处理float的能力差，而Layernorm和Attention等算子不可避免要使用浮点运算

三个应对技术（贡献）：

  1. Chunk-sharing graphs：对应第一个问题，也即做动态shape。思路是预编译几个固定形状的subgraph。

  2. shadow outlier execution：对应第二个问题。NPU直接采用朴素的量化方法，同时，根据稀疏性的思想，把出现激活值的子tensor打包出来分给CPU和GPU计算。

  3. Out-of-order subgraph execution：chunk可以进行乱序执行。设计一个动态调度器，减少推理过程中关键路径的执行时间（也即减少NPU的负载）。

Background

  1. NPU具有优秀的INT8运算能力，（非常）缺乏浮点运算能力

  2. 手机平台CPU，GPU，NPU都在一个SoC上，共享相同的物理内存

  3. 静态shape下，NPU推理DNN的流程图，每次都要重新build和optimize开销最大

  4. 强调Gaps between LLMs and Mobile NPUs：缺少动态shape（截至2024.8，Intel的OpenVINO还没有NPU的dynamic shape算子支持，MindSpore框架对ascend的npu提供了实验性API支持）本文的前提是目前NPU的算子都必须现场编译一次；离群值的问题，NPU不能对整个tensor分组量化，如果拆分成多个tensor量化又导致了性能损失（不能充分利用NPU性能）；LLM中存在NPU无法高效完成的浮点运算算子。

Overview

主要是三个部分，一个是把prompt分成chunk，另一个是对NPU执行scalar统一的量化做矩阵乘，结果和由CPU/GPU计算的离群值加在一起以避免模型性能下降，最后一个就是动态调度机制，让CPU/GPU和NPU的计算尽可能重叠。

图例中chunksize=32，也即32个token为一组执行。a是朴素执行，如果这样的话，prompt的形状只要发生变化，就需要重新编译所有算子的kernel，带来大量的开销。

chunk-share指的是对于一些算子，所有chunk的输入形状都相同，可以共享kernel（例如求QKVO，MLP中的所有算子），无需为每一个chunk都编译一个算子在内存中驻留。

而诸如attention等算子，后续的chunk对前面的chunk存在依赖（例如32-63号token的chunk需要计算0-31号token的注意力得分，因此输入形状是64，64-95号的输入形状是96，以此类推）因此chunk之间无法共享kernel。

使用CPU/GPU计算离群的激活值。量化的离群值问题已经老生常谈，主要就是要避免截断激活值中特别大的值，以免造成性能损失。使用CPU/GPU在NPU计算的同时计算不经过截断的离群值， 则可以几乎无损的快速运算。一个问题是NPU的内存空间是独立的，CPU无法直接访问。拷贝两份权重的开销是不可接受的，因此需要在计算时extract一部分输入，计算这个impact tensor的结果，这样CPU端也不需要全部的权重了。

LLM.INT8()等先前的工作已经观察到离群值是具有稀疏性的，而且在特定的层中，离群值总是出现在特定的一些channel中。因此这部分hot channel的权重驻留在CPU可以访问的内存中。如果推理中遇到了一些输入的离群值对应的权重不在内存的时候，则去disk里加载。

乱序执行。上如是naive的顺序执行，由于涉及到NPU和CPU/GPU的并行计算，这两部分计算不可能是绝对对等的，经常会有一个设备处于空闲。一个设备上Ci-Graphj执行完之后可能会遇到另一个设备正在计算的是当前设备准备计算的下一个子图的依赖的情况。为了减少bubble，这种时候可以在候选队列里选一个可以执行的子图乱序执行。

调度策略具体来说：If subgraph g is to be executed on the CPU/GPU, let S be the set of new subgraphs that can be executed after g is completed. S will be executed on the NPU. A longer execution time of S is beneficial for reducing NPU stalls. Thus, g’s contribution is defined as the total execution time of S. Conversely, if g is executed on the NPU, a shorter execution time of S is beneficial, with the negative value ofS’s execution time as g’s contribution

意思就是永远选择“执行完之后可以解锁总时长最多的NPU子图执行的CPU/GPU子图”

和“执行完之后可以解锁总时长最少的GPU/NPU子图执行的NPU子图”。这样做的原因是NPU总是比CPU/GPU的负载要重，所以应该总是让NPU忙碌，能调度上来的都调度上来，避免因为调度导致NPU的子图被依赖关系卡住无法执行。（示例图中的CPU执行时间似乎更长，和文字描述略有不同）

问题

文章主要解决的是NPU缺乏动态shape、分组量化、还有CPU/NPU共享内存的问题。NPU底层原理上前两点应该不难解决，共享内存可能涉及到CPU的缓存一致性问题，要复杂一些（大概？

目前似乎也不是所有NPU都不支持动态shape，如果过段时间NPU基础设施建设的更好了，是否文章中的优化手段效果就没那么明显了？
